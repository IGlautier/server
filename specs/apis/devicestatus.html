Introduction

This API uses a kind of tree model to allow developers to get at the various bits of data about things on a device.

Aspect
An aspect is an abstract nameable thing (e.g., a "Device", "Battery", "Display", etc.) that the user agent knows about and can interact with. An aspect can contain one or more components.

Component
A component is a concrete instance of an aspect (e.g., a real battery in the device, the real screens of the device). As devices can contain more than one instance of the thing being referred to by an aspect (e.g. two screens, three batteries, two SIM cards), each real instance of an aspect is referred to as a component. Each component has its own unique component name, which is assigned by the user agent and serves as the component's identifier for the life of the application (e.g., "screen_1", "battery2", "os", "hdd", "SDCard").
This specification reserves two special components names ("_active" and "_default") which a user agent MUST NOT use when generating.

Property
A property of an component is something that can be queried to return data (e.g., how much charge a battery has, if the WiFi is connected or not, the SSID of a WiFi network).

A user agent supports aspect X if the user agent can verify that the aspect (hardware or software) exists on the device and/or has not been disabled (either by policy or by a feature declaration).

A user agent supports aspect X and property Y if the user agent supports the aspect and is able to query property Y to retrieve its data.

To make the Device Status API easier to work with, this specification reserves two special component names ("_default" and "_active"). These two component names can be used by a developer in the API as proxies for either a default component or active component.


Default component
The component that is the default component for a given aspect, which may or may not be the active component.
In ECMAScript, a developer uses the string "_default" to refer to the default component.

Active component
The component that is the active component for a given aspect, which may or may not be the default component.
In ECMAScript, a developer uses the string "_active" to refer to the active component.

Code example
  <widget xmlns="http://www.w3.org/ns/widgets"> 
  <feature name="http://picosec.org/api/devicestatus"/> 
  </widget>

The above example shows how a developer declares their intention to access all aspects of the Device Status API.

Code example
  <widget xmlns="http://www.w3.org/ns/widgets"> 
  <feature name="http://picosec.org/api/devicestatus/deviceinfo"/>

The above example shows a developers intention to access the device information aspects of the Device Status API.

Interfaces and Dictionaries

Interface DeviceStatusManager

This interface offers methods to get information about the device status and subscribe to receive notifications of device status changes. To allow for asynchronous of changes to the various properties of components, the user agent keeps a list of active watch operations. Each entry in this list is identified by a number, which is unique within its list for the lifetime of the application.

WebIDL
[NoInterfaceObject]

interface DeviceStatusManager : Service {



    DOMString[]? getComponents(DOMString? aspect);





               

    boolean isSupported(

               DOMString aspect,

               optional DOMString? property);





    PendingDeviceStatusOp getPropertyValue(Function  successCallback,

                                      ErrorCB? errorCallback,

                                      PropQuery query);









    long watchPropertyChange(Function successCallback,

                             ErrorCB? errorCallback,

                             PropQuery query,

                             optional WatchOptions? options);

                             



    void clearPropertyChange(long watchid);

  };

The DeviceStatusManager interface is a discoverable interface through the picosec Discovery API findServices() method and thus inherits the Discovery API Service interface.

Code example
 // deviceStatusService is the device status service, discovered through the picosec Discovery API, that has been selected 
 // by the user
 deviceStatusService.getPropertyValue(
 function(value) {
         console.log("The battery level is at " + value + "%");
 },
 null, {
         property: "batteryLevel",
         aspect: "Battery"
 });

Methods

getComponents

The getComponents() method provides means to get the components of an aspect.

Signature
DOMString[] getComponents(DOMString? aspect);

When the getComponents() method is invoked, the user agent MUST run the steps to retrieve the components.

The steps to retrieve the components are given by the following algorithm. The algorithm returns an array of component names, which may be empty.

  1. Run the general invocation checks algorithm. If no exceptions were generated, continue.
  2. Let components be an empty array.
  3. If the first argument (aspect) null, return components and terminate this algorithm.
  4. Let trimmed aspect be the result of trimming any space characters from the front and back of aspect.
  5. If trimmed aspect is an empty string or does not exactly match the name of a supported aspect, return null and terminate this algorithm.
  6. If trimmed aspect exactly matches the name of a supported aspect:
    6a. For each component of the matching aspect, create a DOMString representation of the component name and add it to the components array.
  7. This offers methods to get information about the device status and subscribe to receive notifications of device status changes. To allow for asynchronous notification of changes to the various properties of components, the user agent keeps a list of active watch operations. Each entry in this list is identified by a number, which is unique within its list for the lifetime of the application.
Code example
//displays contain, for example, ["screen1"] 
var displays = deviceStatusService.getComponents('Display');
console.log("Your device has " + displays.length + " displays");

//iterate through and show the name of each component
for (var i = 0; i  displays.length; i++) {
        var display = displays[i];
        console.log("Display component " + i + ": " + displays[i]);
}

    • Optional: No.
    • Nullable: Yes
    • Type: DOMString
    • Description: for component to retrieve
Return value
components
isSupported

The isSupported() method checks if an aspect is supported and, optionally, if a property that would correspond to a component of that aspect is supported (e.g., a Battery aspect's batteryLevel property).

Signature
boolean isSupported(DOMString aspect, optional DOMString? property);

When invoked, the user agent MUST run the steps to check if an aspect is supported.

The steps to check if an aspect is supported are given by the following algorithm. The algorithm returns either true or false.

  1. Run the general invocation checks algorithm. If no exceptions were generated, continue.
  2. Let trimmed aspect be the result of trimming all white space from the front and back of the first argument.
  3. If the user agent does not supports aspect trimmed aspect, return false and terminate this algorithm.
  4. If the second argument (property) was passed, and it's not null,
    4a. Let trimmed property be the result of of trimming all white space from the front and back of the second argument.
    4b. If trimmed property is a supported property of trimmed aspect, then return true and terminate this algorithm.
  5. Return false.
Code example
if (deviceStatusService.isSupported('Battery', 'batteryLevel')) {
        console.log("batteryLevel property is available");
} else {
        console.log("batteryLevel property is not available");
}

getPropertyValue

The getPropertyValue() method asyncrhonously attemps to read the value of a given component. However, if no component is given, the user agent will default to the active component or the default component of a given aspect.

Signature
PendingDeviceStatusOp getPropertyValue(Function successCallback, ErrorCB? errorCallback, PropQuery query);

The steps to get a property value are given by the following algorithm:

  1. Run the general invocation checks algorithm. If no exceptions were generated, continue.
  2. If either the aspect member or the property member is not present in the third argument (query):
    2a. If the second argument (errorCallback) is not an ECMAScript function, then terminate this algorithm skipping all steps below.
    2b. Let exception be a new DOMException of type "TypeMismatchError", with message "Required aspect or property was not passed." (or similar).
    2c. Queue a task to invoke the errorCallback with exception as the argument.
    2d. Terminate this algorithm, skipping all steps below.
  3. Let aspect name be the result of trimming the value of the aspect member from the third argument (query).
  4. If aspect name does not match an enabled aspect:
    4a. If the second argument (errorCallback) is not an ECMAScript function, then terminate this algorithm skipping all steps below.
    4b. If aspect name is one found in all aspects, but was disabled by policy or via a feature string, then:
    4b1. Let exception be a new DOMException of type "SecurityError", with message "Access to this component was blocked by policy." (or similar).
    4c. If aspect name is not one found in the list of all aspects, then:
    4c1. Let exception be a new DOMException of type "NotFoundError", with message "Aspect could not be found." (or similar).
    4d. Queue a task to invoke the errorCallback with exception as the argument.
    4e. Terminate this algorithm, skipping all steps below.
  5. Let prop name be the result of trimming the value of the property member from the third argument (query).
  6. If the component member is present in query, let component name be the result of trimming component member.
  7. If the component name is not component of the aspect aspect name (e.g., {aspect:"Battery", component: "screen1"}:
    7a. If the second argument (errorCallback) is not an ECMAScript function, then terminate this algorithm skipping all steps below.
    7b. Let exception be a new DOMException of type "InvalidAccessError", with message "Component is not an instance of aspect." (or similar).
    7c. Queue a task to invoke the errorCallback with exception as the argument.
    7d. Terminate this algorithm, skipping all steps below.
  8. If the component member is not present in query:
    8a. If the aspect aspect name has an active component, then let component name be "_active". Otherwise, let component name be "_default".
  9. If the component component name does not have the property property name (e.g., {aspect:"Battery", component: "_active", "fooBar"):
    9a. If the second argument (errorCallback) is not an ECMAScript function, then terminate this algorithm skipping all steps below.
    9b. Let exception be a new DOMException of type "NotFoundError", with message "Property not found in component." (or similar).
    9c. Queue a task to invoke the errorCallback with exception as the argument.
    9d. Terminate this algorithm, skipping all steps below.
  10. Let task be a task that does the following:
    10a. Let value be the value of property property name of the component component name.
    10b. Invoke the successCallback with value as the argument.
  11. Let op be and object that implements the PendingDeviceStatusOp interface.
  12. Associate task with op, so that if op's cancel() method is invoked, it attempts to remove task for the callbacks task queue.
  13. Queue a task.
  14. Return op.
Code example
 //The "_default" component is implied.
 var options = {
         aspect: "Battery",
         property: "batteryLevel"
 };
 
 deviceStatusService.getPropertyValue(
 function batteryQuery(value) {
         console.log("Battery is at " + value + "%");
 },
 null, options);

watchPropertyChange

The watchPropertyChange() method allows the developer to watch for changes to the properties of components.

Signature
long watchPropertyChange(Function successCallback, ErrorCB? errorCallback, PropQuery query, optional WatchOptions? options);

The steps for watching property changes is given by the following algorithm:

  1. Run the general invocation checks algorithm. If no exceptions were generated, continue.
  2. Let subscription id be a user-agent-defined (random or sequential) unique integer that will represent the looping action to be created by invoking this method.
  3. Add subscription id to the list of active watch operations.
  4. Return subscription id, and then continue running this algorithm asynchronously.
  5. Let interval be the value 0.
  6. If the fourth argument (options) is present:
    6a. If the minChangePercent member is present, and it is a number greater than 0, then let percent interval be the value of minChangePercent.
    6b. If the minNotificationInterval member is present, and it is a number, then let interval be the value of minNotificationInterval.
    6c. If the maxNotificationInterval member is present, and it is a number, then let max interval be the value of minNotificationInterval.
  7. If interval is less than a user-agent-defined value for a the smallest amount of time between intervals (i.e., to protect the component from being flooded with requests or because it would drain too much battery, etc.), set interval to that user-agent-defined value.
  8. If max interval is less than interval, discard max interval.
  9. Let old value be the value 0.
  10. Wait, at a minimum (optionally more time if needed), the amount of time indicated by the interval. If the max interval was not discarded, and the time waited exceeds the max interval, do this step again (i.e., wait).
  11. If the entry for subscription id has been cleared from the list of active watch operations, terminate this algorithm. If subscription id has not been cleared from the list of active watch operations, create a task to run the getPropertyValue() method using successCallback as the first argument, and, if present, errorCallback as the second argument, and query dictionary as the third argument. Discard the resulting PendingDeviceStatusOp object.
  12. Factor in the minChangePercent change value. Only queue the task when the change between the old value and new value is minChangePercent or greater.
  13. Queue the task.
  14. Return to step labelled wait.
Code example
 function propertyChange(value, ref) {
         console.log("New value for " + ref.property + " is " + value);
 }
 
 deviceStatusService.watchPropertyChange(propertyChange, null, {
         property: "batteryLevel",
         aspect: "Battery"
 });
 
 //The "_default" component is implied.

         property: "batteryLevel"
 };

 function batteryQuery(value) {
         console.log("Battery is at " + value + "%");
 },
 null, options);

clearPropertyChange

Unsubscribe from notifications for property changes set up by watchPropertyChange.

Signature
void clearPropertyChange(long watchid);

If a valid watchid argument is passed that corresponds to a subscription already in place, then the watch process MUST immediately terminate and no further callbacks MUST be invoked. If the watchid argument does not correspond to a valid subscription, the method should return without any further action.

The PendingSensorConfigOp interface describes the object that is returned by the asynchronous device status method. It makes it possible to abort this operation to a stop if it hasn't produced a result within a desired time or before a given event, thereby possibly reclaiming resources.

Methods

cancel

Method Cancel

Signature
void cancel();

Cancel the pending device status asynchronous operation. When this method is called, the user agent must immediately bring the operation to a stop and return. Allocated resources should be released and any ongoing related network operations should be terminated. An error callback is issued with the DOMError name "AbortError" is issued.

Processing feature declaration

A user agent processes feature elements in accordance with the W3C's Widget Packaging and XML Configuration specification.

If the feature-list contains any features whose feature-name exactly matches a device status feature declarations , then the user agent MUST apply the steps for processing a feature declaration.

The steps for processing a device status feature declaration is given by the following algorithm. Only the first feature declaration is respected by the user agent:

  1. Let declaration be the first device status feature declaration that was present in the configuration document, ignoring all others.
  2. Implement the DeviceapisDeviceStatusManager interface when the widget's start file is instantiated:
    2a. For each supported aspect, identify the components and assign each of them a component name, and, where there are more than one component, identify which is the default and track which components are active. It is recommended that the component name assigned by the user agent to a component be something that is "developer-friendly": i.e., that logically relates to the aspect the component represents (e.g., candidates include "screen_1", "battery2", "os", "hdd", "SD-Card").
  3. If required, disable any aspects as needed by the particular feature declaration (see each device status feature strings).
  4. ignore all other subsequent device status feature declarations.
